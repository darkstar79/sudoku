First tagged release (in preparation). Adds the custom-puzzle feature suite on top of the generated-puzzle baseline.

### Added — Platform support

- **macOS build support** — the application now builds and runs on macOS 13+.
  App bundle (`sudoku.app`), `Info.plist`, and a DMG packaging job are provided.
  Application data is stored in `~/Library/Application Support/Sudoku/`.
  Universal Binary (arm64 + x86_64) is the target for CI/packaging; ARM SIMD
  optimisations are not included in this release (scalar fallback used on Apple
  Silicon). Code signing and notarisation are deferred until distribution begins.
  The `build-windows` CI job is now on-demand only (`workflow_dispatch`),
  consistent with the new `build-macos` job.

### Added — UI

- **Keyboard input for value / pencil / color, with in-app discovery.** A digit acts on the active input mode (Normal → value, Notes → pencil, Color → color); modifier overrides reach any layer from any mode — `Ctrl`/`Cmd`+digit enters a final value, `Shift`+digit toggles a pencil mark, `Alt`/`Option`+digit (1–6) applies a color — and pressing the same digit again undoes it. Erase follows the same rule: `Ctrl`+`0`/`Delete` clears the value, `Alt`+`0`/`Delete` clears the color, `Shift`+`Delete` clears all pencil marks; `Space` cycles the input mode. The shortcuts are surfaced three ways: the mode button now shows the `Space` cycle key on its face, a new **Help → Keyboard Shortcuts…** dialog lists the full map, and a status-bar hint shows `Ctrl=value · Shift=pencil · Alt=color`. All key combos render per-OS (`⌘/⌥/⇧` on macOS, `Ctrl/Alt/Shift` elsewhere) and every label is translatable. The legacy `A`/`B` color keys and the `N` jump-to-Notes key are retired (covered by `Alt+1`/`Alt+2` and `Space`), keeping the gameplay keyboard layer language-neutral.
- **Session timer (right of status bar)** — optional indicator showing total time since the app launched, intended as a reminder for long sessions. Toggle via Settings → Display → *Show session timer*. Off by default. Independent of the per-puzzle timer on the left; keeps ticking through menus and pauses. Persists as `display.show_session_timer` in `~/.local/share/sudoku/settings.yaml` (downstream packagers: new YAML key, default `false`, gracefully ignored by older binaries).

### Added — Custom puzzles

- **Paste-string import** — paste an 81-character puzzle string (digits + `.`/`0` for empty) into the import dialog. Input is validated, length-checked, capped at 4 KiB, and rejected if it has multiple solutions.
- **Edit mode** — enter givens manually via `Ctrl+E`, commit with the toolbar's *Done Editing* action. Validation, uniqueness, and difficulty rating run at commit time.
- **Copy puzzle as text** — `Ctrl+Shift+C` writes the current puzzle's givens to the clipboard as an 81-character string (companion to paste-import).
- **Find step by technique** — pick a specific solving technique (e.g. X-Wing, Naked Triple) and the solver returns the next step in the current board that uses it. Read-only; does not consume a hint credit.
- **Automatic difficulty rating** for imported and edited puzzles. Generated puzzles already arrive pre-rated; imported/edited puzzles now do too, populating the same UIState fields (`puzzle_rating`, techniques, requires-backtracking) so statistics and save files capture an accurate rating.

### Changed

- **Repository renamed** from `sudoku-cpp` to `sudoku` (GitHub URL is now `https://github.com/darkstar79/sudoku`; the old path redirects). Downstream packaging recipes that fetch source by URL should switch to the new path.
- **Reverse-DNS app ID** changed from `org.sudoku_cpp.Sudoku` to `io.github.darkstar79.Sudoku` (follows Flathub naming convention for GitHub-hosted projects without a custom domain, derived from the new repo name). The `.desktop`, AppStream metainfo, and SVG icon files are renamed in lockstep; downstream `.spec` and packaging recipes that hardcode the old paths or `Icon=` value need to be updated.
- **Bundled dependencies bumped** for the Flatpak module set and the Conan recipe used by the Windows build: libsodium `1.0.18` → `1.0.21`, yaml-cpp `0.8.0` → `0.9.0` (Conan recipe was already at parity-via-Flatpak for yaml-cpp; both now align).
- **fmt bumped from `11.2.0` to `12.1.0`** in the Conan recipe (`conanfile.py`). On macOS, Qt6 from Homebrew adds `/opt/homebrew/include` as a system include path, which exposes `fmt/12.1.0` headers to the compiler; linking against a Conan-built `fmt/11.2.0` then produced `fmt::v12` unresolved symbols. The bump aligns the Conan dependency with what Homebrew ships, and is benign on Linux/Windows (spdlog 1.15.3 is API-compatible with fmt 12.x via `SPDLOG_FMT_EXTERNAL`). Downstream OBS/Flatpak packagers that provide their own `fmt` are unaffected.
- **Save file format** bumped from 1.0 → 1.1. A new `PuzzleOrigin` field records whether a puzzle was generated, paste-imported, or entered in edit mode. Loading older 1.0 saves remains supported; missing field defaults to `Generated`.
- **Save files now carry a `rating_model_version` key** (additive; schema stays at 1.1). It records which rating-model version produced a save's stored rating/difficulty so the rating model can evolve in future releases without ever silently reclassifying an existing save: on load the stored rating is always kept as-is (never recomputed), and a save rated by a different model version is flagged stale for an optional, user-triggered re-analysis. The save loader also now reads the schema `version` key (previously write-only); unrecognized future versions load best-effort. Older saves without the new key load unchanged (treated as model version `0`). Downstream packagers: new optional YAML key, gracefully ignored by older binaries.
- **Three advanced-technique difficulty ratings corrected** for closer Sudoku Explainer alignment: Empty Rectangle 4.5 → 4.3, ALS-XZ 7.5 → 6.8, Sue de Coq 7.5 → 6.6. As a result, a newly generated, imported, or edited puzzle whose hardest required technique is **ALS-XZ** or **Sue de Coq** is now labelled **Expert** instead of **Master** (both corrections cross the 7.5 threshold); the Empty Rectangle change stays within Hard and reclassifies nothing. **Existing saved games are unaffected** — a save's stored rating and difficulty are snapshot-preserved on load and never recomputed, so no save is silently reclassified; the rating model bumps from version 1 to 2 (`rating_model_version`), and saves rated by the older model are flagged stale for optional, user-triggered re-analysis. Downstream packagers: no install-tree impact; newly written saves carry `rating_model_version: 2`.
- **Difficulty rating is now context-sensitive per step** — a technique's Sudoku Explainer rating can vary with the individual step rather than being one flat value per technique. The first technique to use this is **Hidden Single**, which now rates **1.2** when the digit is confined to a 3×3 block (SE's easier "block" case) and **1.5** when it is confined only to a row or column, matching Sudoku Explainer v1.2.1. Both values sit inside the **Easy** band (`[0, 2.5)`), so **no puzzle changes difficulty bucket** and **no saved game is reclassified** (stored ratings are snapshot-preserved on load, never recomputed, exactly as for the corrections above). The rating-model version is intentionally **left at 2** — this is a pre-1.0.0 model refinement with no shipped saves to protect. Downstream packagers: no install-tree, save-format, or `rating_model_version` impact.
- **Solver gains budget-bearing overloads** — `solvePuzzle(board, std::chrono::milliseconds)` and a technique-targeted dispatch. The backtracking fallback now honors the deadline so adversarial inputs cannot hang the UI.
- **Hint message clearing** — stale `hintMessage` is cleared on every `getHint` exit path so dismissed hints don't reappear after subsequent moves.
- **Session lifecycle** — entering edit mode finalizes any active statistics session as abandoned (previously orphaned).
- **Language selector** now discovers installed translations automatically. Packagers can ship a new locale by adding `sudoku_<code>.qm` to the translations directory and a matching `.ts` entry in `CMakeLists.txt`; no C++ change is required. Locale codes from `settings.yaml`, the legacy `language.txt`, and the Settings dialog are validated against a BCP 47 subset (`^[a-z]{2,3}(_[A-Z]{2})?$`) before being interpolated into a filesystem path.

### Removed

- **Manual "Analyze Difficulty" menu action.** Auto-rating makes it obsolete — every playable puzzle is now rated at creation time (or loaded with its rating from a save). Retry would have been a placebo anyway since `scoreDifficulty` is deterministic on a given board.

### Security

- **Save-file decompression is now capped at 64 MiB.** Previously the zlib decompression loop could allocate up to ~4096× the compressed input before giving up — a crafted 10 MiB save file could demand ~40 GiB of RAM and crash the process via OOM. The cap is far above any realistic save (YAML saves are typically <1 MiB) but small enough to keep the decompressor's memory bounded; oversized inputs are now rejected with `CompressionError`. Affects both regular save loading and `importSave`. (#12)

### Fixed

- **Applying a cell color now updates the board immediately.** Cell colors are rendered but were omitted from `GameState::operator==`, which `Observable<GameState>::update()` uses to decide whether to notify. A color-only change therefore compared "equal", no notification fired, and the board did not repaint until the next unrelated change (a value entry, selection move, etc.) — so `Alt`+digit and plain Color-mode coloring looked like they did nothing. Colors are now part of the equality, so a color change notifies observers and the board repaints at once. Colors remain ephemeral (not saved, not undoable).
- **Master-difficulty saves now load correctly.** Previously the save loader's range check hardcoded `MAX_DIFFICULTY = 3` (Expert), so every Master save (difficulty 4) was silently rejected with `InvalidSaveData` — permanent data loss for Master-difficulty games. The range now tracks the `Difficulty` enum end-to-end. Existing saves that would not load before this fix will load after it. (#11)
- **AVX-512 dispatch now requires the BITALG sub-feature.** The solver's AVX-512 path issues `_mm512_popcnt_epi16` (in `findMRVCell`), which requires AVX-512 BITALG — present on Zen 5+ and Ice Lake-SP+, but missing on Skylake-X / Cascade Lake / Zen 4. Previously the dispatch gate only checked AVX-512F + AVX-512BW, so these CPUs were selected onto the AVX-512 path and crashed with SIGILL on first popcount. Detection now reads CPUID(7,0).ECX bit 12 and dispatch falls back to AVX2 (or scalar) when BITALG is absent. (#17)
- **Training-mode statistics now persist for all 54 techniques.** The reverse-lookup loop in `training_statistics_serializer` walked enum values `0..GroupedXCycles` (41) and silently dropped the 12 strategies added at 42–53 on reload (SashimiXWing/Swordfish/Jellyfish, Unit/Region ForcingChain, MutantFish, KrakenFish, ALSChain, JuniorExocet, UniqueLoop, ContinuousNiceLoop, GroupedNiceLoop). Mastery progress for the most advanced techniques was lost across launches. The loop now iterates `kAllTechniques` (the canonical list in `training_learning_path.h`), so future strategy additions are picked up automatically. (#13)
- **MainWindow no longer leaves a dangling subscription on the SettingsManager.** A direct `subscribe(...)` discarded its unsubscribe handle; because `settings_manager_` is a `shared_ptr` held by DIContainer and outlives `MainWindow`, the next `setSettings(...)` could invoke a lambda dereferencing a destroyed `this` (use-after-free on `auto_save_timer_`, `training_mode_action_`, etc.). The subscription is now routed through `observer_` (CompositeObserver) so teardown happens in `~MainWindow`. (#14)
- **AVX-512 ops now use aligned loads to match the alignas(64) contract.** `SIMDConstraintState` is `alignas(64)` and its `candidates` array sits at offset 0, but the AVX-512 entry points (`findMRVCell`, `findNakedSingle`) called `_mm512_loadu_si512` (unaligned). GCC has been observed substituting `vmovdqa64` (aligned) once it infers alignment from surrounding context — an intermittent Release-only SIGSEGV waiting to fire if any caller ever passed an under-aligned buffer. The path now uses `_mm512_load_si512` (matching the AVX2 path's `_mm256_load_si256`) with an `assertSimdAligned()` guard at every entry, plus a Catch2 regression test exercising the AVX-512 entry points on boundary inputs. (#18)
- **"Show conflicts" / "Show hints" toggles and status messages are no longer wiped on every keystroke.** `GameViewModel::updateUIState()` was rebuilding a fresh `UIState` and forwarding only `input_mode` / `notes_filled`, silently resetting `show_conflicts`, `show_hints`, and `status_message` to their defaults on every call (which fires on every move, undo, and completion check). Disabling conflict highlighting via Settings would un-disable itself the moment the user typed anything, and the completion message from `checkSolution` vanished on the next move. The method now mutates the existing state in place so sticky fields survive. (#15)
- **Observable is now safe against reentrant `set`/`update` and against subscribers that unsubscribe during their own callback.** Two coupled hazards in `core::Observable`: (1) the interface-observer notification loop iterated `observers_` directly, so an `IObserver::onNotify` that called `unsubscribe()` (which uses `std::erase_if` on the same vector) invalidated the loop iterator → UB on the next step (callback path was already snapshot-protected; interface path was not); (2) a callback that reentrantly called `set()` mutated `value_` before the outer notification finished, so later subscribers in the same batch silently received the newer value instead of the one that triggered notification. Combined with a callback that destroyed another subscriber (or its parent `View`), this manifested as out-of-order notifications and use-after-free on `[this]`-capturing closures. The fix snapshots both `observers_` and `value_` at the top of `notifyObservers`, gates each invocation on live subscription state (mirroring the callback path's existing guard), and defers reentrant `set`/`update` into a `pending_value_` slot that is drained after the outer notification completes — repeated reentries coalesce to the latest value. Observable's header now also documents the subscriber-lifetime contract. (#16)
- **"Undo to last valid state" no longer silently leaves a wrong move on the board after a redo-branch was truncated.** `recordMove` truncates the redo tail when the user makes a new move after undoing, but did not clamp `last_valid_state_index_` to the new array end. If the tracked index dangled past the truncated tail, `undoToLastValid` saw `move_history_index_ < last_valid_state_index_`, exited its loop immediately, and reported "Undone to last valid state" while leaving the wrong move on the board. The index is now clamped at truncation, and the undo loop walks back one move at a time (reverting non-hints, stepping past hints) instead of delegating to `undo()` — which could decrement past the target by `1 + N` for `N` contiguous hints. (#15)
- **Kraken Fish steps no longer carry finned-fish eliminations under the wrong technique label.** `tryFinnedPattern` was bundling in-fin-box finned-fish eliminations (`free_elims`) with chain-verified Kraken eliminations into a single SolveStep tagged as Kraken Fish. Because FinnedXWing/Swordfish/Jellyfish run earlier in the strategy chain and already catch those in-box eliminations, the bundled output (1) inflated Kraken's apparent difficulty contribution, (2) made the step's `position` point at a non-Kraken cell, and (3) confused the technique-correctness harness. The step now reports only the chain-verified Kraken eliminations; the in-fin-box ones are left to the finned strategies. (#20)
- **A Kraken Fish whose fin placement self-contradicts now reports the simpler fin-exclusion instead of a vacuous elimination batch.** When placing the digit at the fin cell propagates to a contradiction, the old `verifyKrakenElimination` returned `true` *vacuously* for every candidate target — so the step listed a batch of cover-cell eliminations "verified" by a chain that had actually self-destructed. The explanation was misleading (the chain excluded nothing), and the step was rated as a full Kraken Fish (SE 8.5) despite doing the work of a much simpler deduction, double-counting in the puzzle rater. The fin-placement propagation is now hoisted out of the per-target loop and run once; on contradiction the step reports the strictly stronger, honest conclusion — *the fin cannot hold the digit* — as a single elimination on the fin cell, rated at the underlying finned-fish level (X-Wing 3.4 / Swordfish 4.0 / Jellyfish 5.4) rather than 8.5. Still tagged `KrakenFish` (a fin-exclusion is not a distinct technique in the standard taxonomy). Difficulty ratings of puzzles reachable via this pattern may shift accordingly. (#40)
- **Kraken Fish no longer reports an inverted "elimination" of the digit from a cell its own fin-chain places that digit into.** When the fin-placement chain *forces* the digit into a cover-line target cell (outside the fin's box), the per-target test treated a zeroed candidate mask as "the digit was eliminated here" — but `placeInState` zeroes a *filled* cell's mask, so a cell the chain had just *placed* the digit into satisfied the elimination test and was reported as an elimination of that digit: the exact inversion of reality. Removing the digit from the one cell it must occupy is unsound. The test is now gated to cells still empty after propagation, so a target the chain fills with the digit is suppressed entirely (neither eliminated nor placed — a placement would be sound only under the unproven fin hypothesis, while the elimination's own validity rests on the complementary fish-body case where that cell instead *loses* the digit, so nothing can be concluded about it). The genuine cover eliminations the same pattern yields are unaffected. The normal solve loop rarely surfaces this (Kraken Fish is the last logical strategy tried, so cheaper techniques almost always resolve a cell first), but **Find step by technique** invokes the strategy directly — a user who picked Kraken Fish on an affected board could be told to remove the correct candidate. A corrected ground-truth sweep finds the pattern is common on hard boards, not rare. Difficulty ratings of puzzles reachable via this pattern may shift accordingly. (#59)
- **Junior Exocet strategy rewritten to the sound Bird/Champagne definition.** The previous implementation was a placeholder that searched targets in the *wrong* bands (outside the JE band rather than inside it), accepted targets containing as few as one base candidate, never checked that the two targets sat in different cross-lines, never checked that the union of target candidates covered the base set, and skipped the S-cell cover-house invariant entirely — feeding unsound eliminations into the solver loop, the puzzle rater, and the auto-difficulty pipeline. The rewrite enumerates row-based **and** column-based patterns, enforces a 3- or 4-digit base set, requires each target to contain every base digit, demands the two targets sit in different cross-lines and different non-base boxes of the JE band/stack, and validates the S-cell cover invariant (each base digit's S-cells must fit within two cover lines). Eliminations are limited to Bird's rule 1 — non-base candidates in the target cells — for which soundness follows directly from the cover invariant; the more aggressive rules 2-9 (mirror-node, double-JE, JE+) remain deferred. (#19)
- **An unreadable statistics history is no longer silently erased.** When the encrypted session-history file (`game_sessions.yaml`) could not be decrypted — e.g. after a hostname or `/etc/machine-id` change (the key is system-derived), or a partial read — `getAllSessions` treated it as empty and the next flush re-encrypted over it, destroying the entire history. Reads now fail closed, and the flush on exit refuses to overwrite a file it could not read, so the original bytes survive. The Statistics dialog detects the condition and offers to archive the file to `game_sessions.yaml.corrupt-<timestamp>` (the original is moved aside, never deleted) so a fresh history can start. Downstream packagers: a new `*.corrupt-<timestamp>` artifact may appear in the user data directory. (#26)
- **Saves, auto-saves, session history, and aggregate statistics are now written atomically.** These wrote directly over their target file (or, for aggregate stats, via a `rename` that never `fsync`'d), so a crash or power loss mid-write left a truncated, unloadable file — and since auto-save is the crash-recovery path, an interrupted auto-save could destroy the very game state it exists to protect. Writes now go to a `<file>.tmp`, are `fsync`'d to durable storage, then atomically `rename`d into place, so a reader always sees either the previous complete file or the new complete one. All four persistence paths share one crash-safe writer. (#25)
- **Importing statistics no longer corrupts the average-completion-time stat.** `importStats` recomputed each difficulty's average time *after* it had already added the imported game count into `games_completed`, so the local contribution was weighted by the post-merge total and divided by the same inflated total — the average drifted on every import and could never be repaired. The weighted mean is now computed from the pre-merge completion counts of each dataset, so importing identical data leaves the average unchanged and merging different datasets yields the correct weighted average. (#27)
- **Cumulative statistics counters widened to 64-bit.** `total_games`, `total_completed`, `total_moves`, `total_hints`, `total_mistakes`, the per-difficulty `games_played`/`games_completed`, and both win-streak counters were 32-bit `int` and could wrap at 2³¹ for very long-lived players, corrupting lifetime totals. They are now `int64_t`. The `aggregate_stats.yaml` schema is unchanged (YAML integers); existing files load unchanged and older binaries still parse the values within their former range. (#27)
- **A libsodium initialization failure no longer crashes the app on startup.** `EncryptionManager`'s constructor called `sodium_init()` and threw `std::runtime_error` on failure; because `SaveManager` and `StatisticsManager` built one unconditionally and `main()` has no top-level `catch`, an init failure (e.g. no `/dev/urandom` in a sandbox, or a seccomp policy blocking the RNG) terminated the process before the window appeared, with no user-facing diagnostic. `EncryptionManager` is now a stateless static utility that initializes libsodium lazily and reports failure as `EncryptionError::InitializationFailed` through the existing `std::expected` chain, so save/load degrades gracefully (the game still starts) instead of crashing. (#28)
- **`CompositeObserver` no longer unsubscribes unrelated subscribers on teardown.** `CompositeObserver::observe` discarded the per-callback unsubscribe token returned by `Observable::subscribe` and instead registered `Observable::clearSubscriptions()` as its teardown — so destroying one observer (or calling `unsubscribeAll()`) wiped **every** subscriber on that observable, not just its own. With both `MainWindow` and `TrainingWidget` owning a `CompositeObserver`, the first to subscribe to a shared observable would be silently torn down by the second's destruction, with no compile or test failure. `observe` now captures and stores the real per-callback token, so each subscription is released independently. (#30)
- **3D Medusa no longer produces unsound eliminations from non-bipartite coloring components.** The strategy 2-colors each strong-link component with a BFS, and already skipped the *elimination* rules (3-6) when the BFS detected an odd cycle (`is_bipartite=false`) — but the *contradiction* rules (1-2) ran first, unconditionally, on the same arbitrarily-colored nodes. For an odd cycle the A/B partition is a meaningless artifact of BFS visitation order, so "this whole color is false" could eliminate correct candidates, feeding bad eliminations into the solver loop, the difficulty rater, and training mode. The non-bipartite check now gates the contradiction rules as well, so such a component is skipped entirely. (Odd strong-link cycles do not arise on a uniquely-solvable puzzle, so well-formed boards are unaffected; this hardens the transient mid-solve case.) (#22)
- **Grouped X-Cycles now emits Type 2 (strong-strong) placements.** The cycle-closure handled only Type 1 (continuous loop) and Type 3 (weak-weak discontinuity) steps; a strong-strong discontinuity onto a single-cell node fell through with no step built, so the forced placement at that cell was never produced — the technique silently under-performed the non-grouped X-Cycles it claimed parity with, and the solver leaned on more expensive strategies / backtracking instead. The closure now mirrors the non-grouped `buildType2Step`, placing the digit at a single-cell discontinuity (a group node cannot be placed and is rejected). Additive and sound by construction — it reuses the proven non-grouped X-Cycles Type 2 logic with a group-rejection guard, is covered by dedicated regression tests, and the strategy-correctness replay reports no new wrong deductions; no eliminations change. Difficulty ratings of puzzles reachable via this pattern may shift accordingly. (#23)
- **Simple Coloring now applies Rule 2 (Exclusion) on a lone 2-cell conjugate pair.** `SimpleColoringStrategy` skipped every coloring component smaller than four cells, so the "Colour Trap" elimination on a single conjugate pair {X,Y} — where any cell seeing both X and Y can have the digit removed, since exactly one of the pair holds it — was never produced. This was a strict under-elimination: the solver was weaker than it should be, and the cheap Simple Coloring tier being skipped could inflate a puzzle's difficulty rating. The size guard is lowered from `< 4` to `< 2` (a 2-cell component); the contradiction rule (Rule 1) is inert below four cells, so the change only adds sound eliminations. (#23)
- **The `GameCommand` pipeline now honors every verb and gates on real preconditions.** `executeCommand`'s switch handled only 9 of the 16 `GameCommand` verbs — `NewGame`, `SaveGame`, `LoadGame`, `ValidateBoard`, `PauseGame`, `ResumeGame`, and `ClearNotes` fell through to a warning and silently did nothing — while `canExecuteCommand` returned `true` for every unhandled verb, so a button bound to `SaveGame`/`ResumeGame` appeared enabled then no-op'd (e.g. `ResumeGame` reported available with no game loaded). All verbs are now implemented (with a `GameCommandArgs` payload carrying the difficulty / save name / save id for the parameterized verbs), `canExecuteCommand` returns the real precondition per verb and fails closed for unknown values, and `MainWindow` dispatches new-game / save / load / undo / redo / toggle-mode through the pipeline instead of bypassing it. (#30)

### Internal

- New `IPuzzleAnalyzer` interface (`parseString`, `serializeToString`, `validate`, `checkUniqueness`, `scoreDifficulty`) backs all custom-puzzle workflows.
- Localization updated for new strings (English, German).
- Test coverage: 980 unit / 14 integration / 11 UI; lines 88.3% / functions 89.8% / branches 55.5% (all above 80% / 70% / 55% thresholds).
- `requirements.txt` now also pins `cmake>=3.28` and `ninja>=1.11`, so `pip install -r requirements.txt` (or `uv pip install -r requirements.txt`) brings the full Python-installable build toolchain. The Ninja generator referenced by `CMakePresets.json` needs `ninja.exe` on PATH; this removes one manual setup step for new contributors on Windows.
- Windows build scripts now auto-detect the newest installed Qt6 MSVC 2022 64-bit kit under `C:\Qt\6.*` (proper `[version]` sort, so 6.11 ranks above 6.9) via a new `scripts\find_qt6.ps1` helper. Replaces the previous hardcoded version allow-list (which topped out at 6.10.2 and silently failed on any newer release). `QT6_DIR` still overrides for non-default install locations.
- Windows-setup docs in [README.md](README.md) refreshed to lead with a `uv venv` + `requirements.txt` workflow, mention the MinGW-vs-MSVC kit gotcha explicitly, and document the new Qt6 auto-detect path. [docs/CONAN_PROFILES.md](docs/CONAN_PROFILES.md) rewritten to reflect that the Windows build path uses the auto-detected `default` Conan profile (the previously-documented `msvc-release`/`msvc-debug` named profiles were never shipped in the repo and would have confused new contributors).
- **New [docs/PACKAGING.md](docs/PACKAGING.md)** — upstream-to-downstream-packager interface. Documents the install tree, active migrations (the app-ID rename), recent packager-relevant changes (dynamic locale discovery, bundled-dep bumps), Flathub-prep status, and anticipated future changes (planned `sudoku_cpp` → `sudoku` repo rename). OBS recipes for the four `home:AndnoVember` namespaces should reference this doc going forward.
- **Windows build scripts migrated from `.bat` to PowerShell.** All four `scripts\*.bat` files are replaced by three idiomatic `.ps1` scripts: `build_windows.ps1` (parametrized `-Config Release|Debug`, collapses the previous Release/Debug duplication into one script), `run_tests_windows.ps1`, and `create_installer.ps1`. The migration fixes two bugs that affected the old batch scripts: (1) the build now passes `-s compiler.cppstd=23` explicitly to Conan, so dependency builds (Catch2, etc.) stay ABI-compatible with the project's C++23 code regardless of what the user's auto-detected Conan profile defaulted to — previously this caused `unit_tests.exe` to fail to link with `unresolved external Catch::StringMaker<basic_string_view>::convert` on a fresh Windows install; (2) script failures now propagate exit codes correctly via `$LASTEXITCODE` and `throw`, replacing the broken `exit /b %ERRORLEVEL%` pattern in the batch trampolines (which expanded `%ERRORLEVEL%` at parse time and silently exited 0 on any failure).
- **New `SUDOKU_ENABLE_COVERAGE` CMake option** (default `ON`) gates gcov instrumentation on Debug/RelWithDebInfo builds. Default behaviour is unchanged (coverage still on for the coverage build); the option lets non-coverage RelWithDebInfo jobs (e.g. the nightly strategy-correctness replay) build faster without `--coverage`. Packagers building RelWithDebInfo who do not want gcov flags can now pass `-DSUDOKU_ENABLE_COVERAGE=OFF`.
- **CI cache + job-graph hardening.** Actions ccache caches are now keyed by build *flavor* (`rwdi-cov` / `rwdi-nocov` / `rel-san` / `rel-plain`) rather than by job, so same-flag jobs share a cache and different-flag jobs stay isolated; all RelWithDebInfo CI/nightly jobs pin `compiler.cppstd=gnu23` for consistent dependency reuse. `clang-tidy (diff)` now depends on `test-and-coverage` and restores its cache (no redundant build); `i18n-check` is gated on `format-check` and caches Conan packages. A new `cache-cleanup.yml` workflow deletes a PR's caches when it closes, keeping the repo under the 10 GB Actions cache budget.

### Experimental

The following features are compiled in but hidden by default. Enable them under *Settings → Experimental*. They are **not part of the 1.0 stability commitment** — data formats, UI, and behavior may change or be removed in any 1.x release.

- **Training Mode** — interactive technique trainer with 11 categories of progressive hints, mastery tracking, and per-technique statistics. Reached via *Game → Training Mode* once enabled.
- **Progressive coaching hints** — in-game hint that walks you through the next step in three levels of help before applying it, with a check/apply workflow. Reached via *Help → Get Coaching Hint* (Shift+H) once enabled.

### Platform support

- **Linux (Fedora)** — primary development platform; full interactive use by the maintainer.
- **Linux (Ubuntu 24.04)** — built and full test suite run in CI on every push.
- **Windows 11** — installer artifact attached to the release. Built and tested in CI on `windows-2025` (per push to `main` and on demand); smoke-tested by the maintainer on Win11 before tagging.
- **Linux (openSUSE)** — RPM snapshots produced downstream via the openSUSE Build Service (not built directly by the project).
- **macOS** — not supported in 1.0; out of scope.

See [README.md](README.md#platform-support) for the difference between "tested", "CI-built", and "packaged downstream".

### Known limitations

- No user-facing timer pause. The timer runs continuously while a puzzle is open.
- Open sub-dialogs (Settings, Import Custom Puzzle, Find Step by Technique) do not re-translate when the language is changed from within them — close and reopen the affected dialog to see it in the new language. The main window shell (menu bar, toolbar, button panel, status bar) and the Training Mode widget retranslate immediately.
- No theme system; colors are fixed (light theme).
- Save file format v1.1 is the 1.0 stability boundary. Older v1.0 saves still load (the missing `PuzzleOrigin` field defaults to `Generated`). Pre-1.0 snapshot binaries may not load v1.1 saves; downgrading after upgrading is not supported.
